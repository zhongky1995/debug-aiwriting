# Internal Operations Register

Use this for internal execution plans, SOPs, project memos, meeting follow-ups, ownership tables, rollout plans, handoff notes, and operational checklists.

## Reader And Function

The reader needs to know what to do, who owns it, when it is due, what input is required, what output counts as complete, and what happens when the normal path fails.

## Minimum Execution Unit

For each meaningful task, specify as many of these as the source supports:

- owner
- trigger or start condition
- input
- action
- output or deliverable
- deadline or rhythm
- dependency
- acceptance rule
- exception or escalation path

## Register Rules

- Use direct operational Chinese. Prefer precise verbs over strategic labels.
- Keep project terminology only when the team actually uses and understands it.
- Do not turn every task into a full paragraph; tables and checklists are often the correct form.
- Avoid motivational conclusions, capability claims, and client-facing sales language.
- Distinguish required actions, optional suggestions, and unresolved decisions.
- A role name is not an owner if several people share the role and nobody is accountable.

## Common Rewrites

Weak:

```text
加强跨部门协同，确保项目高效推进。
```

Better:

```text
项目经理每周二汇总产品、内容和投放三方未解决事项；超过两天未确认的价格、素材和上线时间，提交业务负责人决定。
```

Weak:

```text
持续优化内容质量。
```

Better:

```text
内容负责人每周抽查 20 条已发布内容，记录事实错误、品牌口径和评论问题；同一问题出现 3 次，下周任务包统一修改。
```

## Final Check

- Can each required action be assigned and checked?
- Are deadlines, inputs, outputs, and dependencies visible where needed?
- Are exceptions and escalation paths stated for fragile steps?
- Does the document remain an internal execution tool rather than becoming a strategy deck or external proposal?
