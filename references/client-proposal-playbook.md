# Client Proposal Playbook

Use this reference when Chinese writing must sound like a credible client-facing proposal, strategy deck, integrated marketing plan, CRM/private-domain plan, KOC/community plan, or sales material.

The goal is not to make a proposal casual. The goal is to remove fake polish and make the client believe three things:

1. You understand the real business question.
2. Your strategy has a mechanism, not just a list of actions.
3. The work can be executed, judged, and adjusted.

Choose presentation density by page role. A strategy title, executive judgment, mechanism diagram, execution table, and narrative explanation should not all sound equally conversational. Use the professional marketing register for deck-like marketing plans.

## Proposal Substance Pass

Before rewriting, extract these six items. If an item is missing, mark it as an assumption instead of inventing it.

1. **Client decision question**: What does the client need to decide after reading this page or section?
2. **Diagnosis**: Where is the current problem located: product, user, channel, content, conversion, retention, execution, data, or organization?
3. **Strategy mechanism**: What chain connects the action to the result?
4. **Execution proof**: What minimum object proves it can run: SOP, role split, timeline, script, checklist, data table, dashboard, review rhythm, or sample asset?
5. **Validation**: Which metric, milestone, or review question decides whether the work is working?
6. **Boundary**: What this plan does not promise, does not solve yet, or needs the client to confirm.

## Mature Proposal Patterns

Use these patterns when source materials support them.

### 1. Goal -> Link -> Resource Position

For growth, conversion, CRM, or platform operation plans, avoid starting from channel actions. Start from the business formula and the link with the largest uncertainty.

Weak:

```text
我们将通过多资源位触达提升转化效率。
```

Better:

```text
先把目标拆到曝光、点击、咨询、入会和复购几个关键段，再看不同人群在不同资源位上的效率差异。后续资源不平均分配，而是按保留、放大、测试、停用四类处理。
```

### 2. User Stage And Channel Role

Do not write "面向用户" as if all users are the same. Name the stage or role.

Useful distinctions:

- 新客 / 高意向 / 已购 / 复购 / 沉睡 / 高认同传播人群
- 认知触发 / 搜索求证 / 咨询转化 / 交易后服务 / 复购召回
- 公域触发兴趣，私域建立信任和推动转化，内容反馈回到下一轮选题

### 3. Public-To-Private-Domain Continuity

For integrated marketing and private-domain proposals, the client needs to see the handoff.

Good proposal language names:

- public touchpoint: content, KOC, search, live stream, community, store, package insert, event
- capture action: comment keyword, private message, form, membership benefit, QR code, consultant invitation
- private-domain action: welcome, questionnaire, segmentation, consultation, group rhythm, follow-up
- business result: first purchase, repeat purchase, high-value user identification, content feedback, resource pool

### 4. Content -> Live -> Community Conversion

For education, knowledge payment, high-explanation products, and trust-heavy categories, content is not just "what to post." It should connect audience, scene, emotion, and stage.

Minimum credible chain:

```text
content task -> live-stream type -> group rhythm -> role action -> 7-day conversion or review metric
```

Useful proof objects include Day0-Day6 rhythm, FAQ, assistant/SOP role split, 1v1 follow-up rule, group announcement, and issue feedback list.

### 5. KOC / KOS / UGC Is A Mechanism, Not A Resource Count

Avoid selling only the number of accounts or posts. Explain the operating chain:

```text
recruit -> onboard -> assign topic pack -> publish -> collect links -> review quality -> incentive -> weekly optimization -> retain high-value accounts
```

Client-facing boundaries matter:

- Do not imply fake users, guaranteed virality, guaranteed algorithm traffic, or fabricated experience.
- If the plan uses UGC/AIGC, state what is generated, what is reviewed, and what cannot be claimed as real usage.

### 6. Execution Proof Beats Capability Claims

In proposal writing, "we can execute" is weak. Replace it with a visible object.

Use combinations such as:

- role action + script + feedback form + process map + metric
- project calendar + weekly meeting rhythm + change log + acceptance criteria
- content task package + review rule + publishing return table
- questionnaire + segmentation rule + follow-up SOP

### 7. Case Evidence Must Prove One Thing

Do not use cases as decoration. Decide what the case proves.

Examples:

- A private-domain case proves that people who bought once can still be found, served, and reminded when they may need to buy again.
- A KOC case proves that real expression needs recruiting, content tasks, link collection, incentive, and review.
- A search/content case proves that content must be findable, understandable, and reusable, not only published.
- A CRM case proves that member growth, visit frequency, repurchase, targeted messages, and member-day campaigns need one clear record of who the user is and what happened before.

## Proposal Language Rules

The examples below demonstrate semantic clarity, not one mandatory tone. Compress them into professional page language when the artifact is a strategy deck; keep fuller sentences when it is a narrative proposal or explanatory memo.

- Prefer "先看清问题卡在哪，再决定从哪一段开始重建" over "提供一站式全域解决方案."
- Prefer "把用户从咨询、成交到复购的记录留清楚，后面该回访、该提醒时还能找得到人" over "打造私域闭环."
- Prefer "按用户阶段和资源位拆分动作" over "精细化运营用户."
- Prefer "本阶段不以单条爆款或直接订单作为验收标准" over "助力品牌声量与销量双增长."
- Prefer "如果群里没人答问题、没人跟进，就不该继续扩大直播流量" over "持续扩大流量入口."

## Rewrite Procedure For Client Proposals

1. Rewrite the page title or section opening as a claim, not a category.
2. Put the client-side problem before the provider capability.
3. Convert abstract benefits into a mechanism with actors and handoffs.
4. Add execution proof only from available materials.
5. Add one boundary if the section could otherwise sound like an overpromise.
6. Remove ornamental confidence words unless supported by data or confirmed facts.

## Common Rewrites

```text
AI-like:
通过全域整合营销矩阵，赋能品牌实现品效合一。

Client-facing:
这套方案不把达人、内容、私域和复购分开做，而是先确定用户从哪里知道产品、在哪里确认自己要不要买、买完后去哪里咨询，再把用户买完后的问题和反馈用来调整下一轮内容和产品判断。
```

```text
AI-like:
打造 KOC 社群生态，持续激活用户声量。

Client-facing:
KOC 不是一次性招募名单。项目要跑通招募、入群、内容任务、发布记录、激励和周会复盘，才能判断哪些用户只是参加了一次，哪些后面还值得继续合作。
```

```text
AI-like:
沉淀私域用户资产，形成长期运营闭环。

Client-facing:
这里说的私域不是把人拉进群就结束，而是按用户状态继续做事：新客先问清楚问题；高意向用户继续答问题、提醒下单；已购用户继续回答使用问题、提醒补货；高认同用户再邀请他们对内容提意见，或参与传播任务。
```

```text
AI-like:
我们将用内容种草提升品牌专业心智。

Client-facing:
内容不先堆产品卖点，而是从用户已经在问的状态、误区和使用场景切入。这样做的目的，是让内容完成三件事：用户搜得到，看完知道问题怎么解决，有咨询意向时知道下一步找谁。
```

## Output Format

Follow the output policy in `SKILL.md`: for clean client-ready copy, put the revised proposal language first and do not expose the reasoning unless the user asks. Use the format below for proposal review, internal checking, or when the user wants the decision logic.

```text
写作判断：
- 客户决策问题：
- 证据边界：
- 改写策略：

改写稿：
[proposal copy]

主要调整：
- [how the rewrite strengthened diagnosis/mechanism/execution/proof]
- [what overclaim or generic language was removed]
```
